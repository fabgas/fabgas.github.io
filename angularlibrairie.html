Construire et partager une librairie

https://medium.com/@jinalshah999/how-to-build-and-publish-angular-library-a333d6ff12c0

Qu’est ce qu’une librairie Angular ?

Il s’agit d’une collection de composants, services, directives, etc qui doivent être partagés dans différents projets Angular.

Plus précisément, en programmation, la Libraries est une collection de routines précompilées qu’un programme peu utilisé.

Une librairie Angular est un projet Angular qui diffère d’une application par le fait qu’il ne peut ête exécuté seule. Une librairie doit être importée et utilisée dans une app. angular.

Donc, Une librairie Angular est un code partageable qui fournit des fonctionnalités réutilisable.

Pourquoi avoir besoin d’une librairie ?

Parfois les composants, services peuvent être appliqués à de multiples projets angular. Pour réutiliser ces bouts de code, le concept de librairie a été inventé dans Angular.

Cette librairie peut être utilisés localement dans votre workspace ou être publié sous format de package npm pour partage avec d’autres projets ou avec d’autres developpeurs Angular.

Création d’une librairie

A l’aide d’Angular CLI il est plus facile pour générer une librairie.

Pour créer une librairie on dispose d’une commande très simple :

ng generate library <<library-name>>

Mais, avant tout créons une application Angular dans laquelle nous crééons une librairie Angular qui sera utilisée localement. Ensuite, cette librarie sera publiée sur un npm local.

Démarrons par la création d’une application Angular classique.

ng new angularLibraryDemo

et maintenant à l’intérieur du projet , on peut ajouter la librairie

ng generate library indicators-lib

(attention, ici indicators-lib est le nom de la library et ce nom doit pas être en majuscules)

De nombreux fichiers sont créés et des fichiers ont mis à jour (update.json, package.json et dsconfig.json dans l’application principale).

La création de la librairie à l’intérieur d’une application permet de vérifier que la librairie fonctionne avant de publier la librairie.

La librairie est créé dans un sous répertoire spécifique

Lors de l’ajout de cette librairie, elle a été déclarée dans le fichier package.json sous le tag projet.

Noter l’indication "library" pour la propriété projectType.

Dans package.json, la dépendance ng-packagr a été ajoutée. Cette librairie gère le packaging de la librairie

Sont stockés ici les chemins pour le build (dist) ou la librairie sera compilée et stockée.

Concentrons nous sur le répertoire de la librairie.

La librairie a son propre fichier package.json ainsi que d’autres fichiers avec leur propre version et dépendances.

Regardons le fichier public-api.ts à l’intérieur de répertoire src

Il s’agit du point d’entrée de la librairie. Quelque soit le composant, services Directives que l’on souhaite exposer à l’extérieur doit être exportée ici. En effectuant cette déclaration, n’importe qui peut utiliser la librairie en l’important.

Maintenant, focalisons nous sur le répertoire lib.

A l’interieur on peut voir que un composant, un service et un module ont été créés. Chacun d’entre eux a été créé dans le fichier public-api.ts.

nommage de la librairie

Lors de la génération de la librairie il est possible d’utiliser un nommage plus complet de la librairie

ng generate library @galma/indic-lib  --prefix=galma

La librairie est alors créer dans une hiérarchie de répertoire .

Le prefix permet de préfixer les selecteurs des composants créer par cli.

Utilisation de cette librairie dans notre application locale

Avant d’utiliser cette librairie il faut la builder. Il n’est pas possible de l’utiliser la librairie sans l’avoir builder dans un répertoire dist.

ng build <<librairie-name>>

Cette commande va créer un répertoire un répertoire dist et à l’intérieur la librairie compilée.

On peut maintenant utiliser la librairie.

Importer la librairie dans l’application angular

La librairie est ajoutée dans l’application de manière classique. Cela débute par l’ajout du module de la librairie dans le module principale.

import { IndicatorsLibModule } from 'indicators-lib'
(...)
imports: [..., IndicatorsLibModule,...]

Les composants et autres directives peuvent être utilisées.

Ajout d’un nouveau composant dans la librairie

Commencons par générer un composant dans la librarie. Il aut se positionner dans le src de la librairie et executer la génération:

ng g c CounterDemo --project indicators-lib

Le module de la librairie a été mis à jour. Cependant, nous voulons utiliser les composants à l’extérieur . Pour cela il faut modifier le module en ajoutant le nouveau composant dans le tableau des exports du module.

mais également il faut ajouter la déclaration de la librairie dans le fichier public-api.ts

Il faut rebuilder la librairie pour quelle soit utilisable

Ajout d’une dépendance dans la librairie

La librairie peut nécessite ces propres dépendances. L’ajout de celles-ci se fait de manière classique.

npm i ng2-charts --save

Mais on obtient un message d’erreur Dependency ng2-charts must be explicitly whitelisted.

ng-package.json
"whitelistedNonPeerDependencies": [
    "."
  ]

Il faut également ajouter la librairie comme dépendance de l’application principale.

Publication de la librairie vers npm

Publication locale

Pour cela on doit builder la librairie en tant que package npm et ensuite la publier dans le registry npm local pour être utiliser dans d’autres projets Angular localement.

Cela signifie deux étapes :

1 Créer le build
2 Créer un fichier pack
3 La première étape consiste à créer le build. Cela a déjà été mentionner précédemment. 
ng build indicators-lib
1 Pour créer un fichier pack, il faut dans un terminal et naviguer vers le répertoire dist de la librairie et executer la commande de création du fichier pack 
npm pack

Cela va créer un fichier Pack avec un nom indicators-lib-0.0.1.tgz

La publication a eu lieu. Il est maintenant possible de créer un nouvelle application en locale et de l’utiliser dedans.

L’installation en tant que package dans cette nouvelle application est cependant ardu car il faut directement référencer le fichier pack

npm install <<path to dist>>/indicators-lib-0.0.1.tgz**

Publication distante (npm)

Pour publier le package sur npm il faut avoir un compte. Il faut s’inscrire sur https://www.npmjs.com/signup

Ensuite il faut s’ajouter comme user localement :

npm adduser

et vérifier que l’on est logger par

npm whoami

Une fois logger il faut aller dans le répertoire dist de la bibliothèque et publier la librairie. Le fichier ayant déjà été construit via npm pack , seule la publication est nécessaire :

npm publish --acess public

La bibliothèque est alors visible à l’adresse https://www.npmjs.com/package/<library-name>;

La librairie est maintenant utilisable depuis n’importe quel poste.

Le nom de la librairie est un peu brute. Il est possible de le changer dans le fichier package.json de la librairie, via le tag name. Cela est d’ailleurs conseillé car le nom de package risque de ne pas être unique.

Utilisation des sous modules git

https://medium.com/dev-friends/once-upon-a-library-1-3-6ce6a714c676

La construction des librairies Angular est propice à l’utilisation des sous modules git. On peut imaginer que l’application principale soit conservé dans un repository A et que les librairies soient stockées chacune de leur repository.

Lié chaque librairie à un repository permet à tout développeur de cloner le repository, modifier la librairie et publier une nouvelle version.

Rappel de la création de librairie

Toute librairie doit être créé à l’intérieur d’une application Angular

ng new once-upon-library --create-application=false (1)
1 l’option create-application permet ou non de créer une application . Pour tester la librairie mieux vaut mettre true…​

Ajouter l’application dans un repository git

git remote add origin git@github.com:fabgas/angularlibrairie.git

Ensuite il faut procéder à la création de la librairie.

ng g library @devfriends/df-library --prefix=df (1)

Il faut lier cette librairie à un repository git différent de celui de l’application principale.

Construction des repository

git submodule add git@github.com:fabgas/angularLibrairieSub.git (1)
            projects/galma/indic-lib (2)
1 correspond au repository dans lequel est stockée la librairie
2 le répertoire contenant la librairie dans l’application principale

Ensuite on peu pusher indépendamment les deux repo. Le repository de l’application principale référence un commit du sous module. Pour maintenir le lien entre les deux repertoires il faut modifier ce lien , en indiquant un nouveau commit. === Mise à jour ===

1 Une modification est faite dans le sous module . Sur la branche suivi elle correspond à un certain commit.
2 Pour correctement faire le lien entre le module et le sous module il faut donc faire un push depuis le répertoire principale de l’application.
Attention les deux repository peuvent travailler sur des branches différentes.

Généralement on ne travaille que dans le sous module, les commits se font uniquement dans ce répertoire. Puis lorsque la librairie est stable, i lfaut alors pousser les modificatios dans le répertoire principale pour référencer le bon commit sur le sous module.

https://www.vogella.com/tutorials/GitSubmodules/article.html

Initialisation d’un repository contenant des sous module

git clone --recursive --jobs 8 [URL to Git repo]

Passer des paramètres de l’application vers la librairies

Je veux passer des paramètres à la librairie (peu importe ces paramètres), comment faire sachant que la librairie ne doit pas connaître l’application. Il doit s’agir de paramètres globaux et constant. D’autres paramètres spécifiques à un composant peuvent être transmis via un @Input. On peut par exemple citer une url que seul l’application connaît.

Librairie

La première étape consiste à construire un objet recevant les paramètres et un service pour les injecter

libConfig.ts : paramètres à injecter
import { InjectionToken } from '@angular/core';

/**
 * Injection de configurations dans cette librairie
 *  url : adresse du serveur
 */
export interface LibConfig {
  url: string;(1)
}
export const LibConfigService = new InjectionToken<LibConfig>('LibConfig');(2)
1 paramètres que l’on pourrait transmettre
2 service permettant d’injecter des informations non décrit dans une instance (inteface).

Dans le module de la librairie, i lfaut déclarer une méthode forRoot(standard) et ajouter une des objets en paramètre

indic-lib.module.ts : fichier du module de la librairie
export class IndicLibModule {
  static forRoot(config: LibConfig): ModuleWithProviders {
    return {
      ngModule: IndicLibModule, (1)
      providers: [
        {
          provide:  LibConfigService, (2)
          useValue: config (3)
        }
      ]
    };
  }
}
1 désignation du module
2 Instanciation du service
3 Paramètres à passer au service lors de son instanciation

l’utilisation de ce service se fait simplement

export class IndicLibComponent   {
  public libConfig: LibConfig;
  constructor(@Inject(LibConfigService) public config) {
    this.libConfig = config;
  }
  public getConfig(): LibConfig {
    return this.libConfig;
  }
}

Application

Enfin, il faut bien fournir les paramètres à la librairie depuis l’application. Lors de la déclaration du module , il suffit de passer les bonnes valeurs :

IndicLibModule.forRoot({
      url: environment.endPoint })

Gestion du style de la librairie

Une librairie angular va nécessairement utiliser un style particulier. Ce style doit être importé dans l’application pour pouvoir s’appliquer.

Si on imagine que la librairie a défini un style dans un fichier "__styles.scss". Ce fichier doit être importé dans l’application si l’on souhaite l’utiliser.

Au niveau du styles.scss, il faut importer le fichier

@import '../projects/galma/indic-lib/src/lib/themes';

ou dans le fichier angular.json avec l’autre style

"styles":
    [
        "projects/galma/indic-lib/src/lib/_variables.scss",
        "src/styles.scss"
    ]

lorsque la librairie est dans le répertoire module, le chemin doit s’adapter.

Dans ce cas on import uniquement les variables (les couleurs par exemple). Ces variables ne peuvent être surchargées!

Les classes définies dans les fichiers peuvent elles êtres modifiés. Par exemple dans la librairie, la classe suivante est déclarée :

.container {
    background-color: $color-red;
}

la surcharge dans l’application peut être faite ainsi

.container {
    background-color: blue !important;
}

(sans l’import ca marche également, ce qui est bizarre)

En réalité, il semble y avoir une phase de compilation en scss qui rend la sucharge complexe.

L’utilisation des variables css(3) permet plus facilement cette surcharge.

Dans la librairie angular, dans un fichier dédiés on déclare un certains nombre variables

:root {
    --color-background : #FFFF00
}

Ensuite cette variable est utilisée dans un composant (classe container).

Dans l’application principale, on importe le fichier contenant les variables.

Dans le fichier styles.scss, il est possible de redéclarer les variables.

style.(s)css
:root {
    --color-back: #ac7070;
}
